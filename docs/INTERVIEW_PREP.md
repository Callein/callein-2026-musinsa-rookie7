## 1. Technical Deep Dive

### 1.1 동시성 제어: "왜 비관적 락이었나?"

> 이 질문은 거의 확실히 나온다. 핵심은 **"낙관적 락을 몰라서가 아니라, 알고도 비관적 락을 선택한 근거"**를 보여주는 것.

**스토리 흐름:**

**[문제 정의]** "수강신청의 본질은 이커머스의 한정 수량 재고 차감과 동일한 구조입니다. 정원 1명 남은 강좌에 100명이 동시에 접근하면, 정확히 1명만 성공해야 합니다."

**[대안 검토]** "네 가지 전략을 비교했습니다."

| 전략 | 기각 사유 |
|------|---------|
| 낙관적 락 | 충돌률이 극도로 높은 수강신청에서 재시도 폭주(retry storm) 발생. 100명 동시 접근 시 99명 실패 → 재시도 → 98명 실패의 연쇄. 시스템 부하 기하급수 증가 |
| Application Lock | `asyncio.Lock`은 단일 프로세스에서만 유효. 수평 확장 시 무력화 |
| Redis 분산 락 | 추가 인프라 의존성. 락 해제 실패 시 장애 전파 위험. 이 규모에서는 오버엔지니어링 |
| **SELECT FOR UPDATE** | **DB 레벨 보장, 단일 행 잠금, 대기 후 확정적 결과** |

**[결정적 설계]** "데드락을 코드가 아니라 API 설계로 원천 차단했습니다. '한 번에 한 과목만 신청'이라는 제약을 두어, 단일 트랜잭션에서 하나의 Course 행만 잠그므로 순환 대기(circular wait)가 물리적으로 불가능합니다."

**[격리 수준]** "Serializable까지 올릴 필요가 있는지도 검토했습니다. `SELECT FOR UPDATE`는 잠금 획득 시점에 최신 커밋 데이터를 읽으므로, Read Committed + 행 잠금으로 충분했습니다. Serializable은 사후 검사 방식이라 고부하에서 오히려 성공률이 극도로 낮아집니다."

**[무신사 연결 — 꼭 덧붙일 것]** "이 패턴은 무신사의 한정판 발매와 동일한 구조입니다. 인기 상품에 수천 명이 동시에 접근할 때, 재고보다 많은 주문이 통과하면 안 됩니다. `SELECT FOR UPDATE`의 '줄 서기' 방식이 이런 시나리오에 가장 적합합니다."

### 1.2 꼬리 질문 대비: "그럼 트래픽이 10배로 늘면?"

> 현재 설계의 한계를 솔직히 인정하되, 확장 전략까지 그려서 성장 가능성을 보여준다.

**답변 구조:**

"현재 설계는 단일 PostgreSQL 인스턴스에서 완벽하게 동작합니다. 서버를 수평 확장하더라도 동일 DB를 바라보는 한 유효합니다. 하지만 진정한 한계는 DB 자체가 분산되는 시점입니다."

**확장 전략 3단계:**
1. **즉시 적용 가능:** 인기 강좌의 잔여 정원을 Redis에 캐싱 → DB 접근 전 사전 필터링으로 불필요한 락 경합 감소
2. **중기:** 수강신청 대기열(queue) 시스템 도입 → 동시 접속 부하 평준화 (RabbitMQ 경험 연결)
3. **장기:** Redis 분산 락(Redlock) 또는 PostgreSQL Advisory Lock 검토 → DB 분산 시점에서 전환

"이미 CMS 프로젝트에서 RabbitMQ 기반 대기열 시스템을 구축한 경험이 있어서, 대기열 도입은 실질적으로 적용할 수 있습니다."

---

### 1.3 꼬리 질문 대비: "배치 삽입에서 기술적으로 어려웠던 점은?"

**답변:**

"10,000명 학생과 500개 강좌를 1분 이내에 생성해야 했습니다. 세 가지 전략을 적용했습니다."

1. **배치 삽입:** 1,000건 단위 묶음 처리로 DB 라운드트립을 10,000회 → 10회로 줄임
2. **결정론적 생성:** `random.Random(42)` 고정 시드로 재현 가능한 테스트 데이터 확보
3. **멱등성:** 서버 재시작 시 기존 데이터 확인 후 스킵

"실제 병목은 bcrypt 해싱이었습니다. 10,000명의 비밀번호를 bcrypt로 해싱하는 것이 시드 생성 시간의 상당 부분을 차지했습니다. 운영 환경에서는 보안상 적절하지만, 테스트 데이터 생성에서는 해시 라운드를 낮추는 것이 합리적이었을 것입니다. 이것은 시간 안에 해결하지 못한 트레이드오프입니다."

> 솔직하게 한계를 인정하는 것이 오히려 신뢰를 준다.

### 1.4 1차 코딩테스트 연결: "defaultdict 설계"

> 면접관이 1차 코딩테스트도 함께 볼 가능성이 높다. 1차와 2차를 관통하는 엔지니어링 사고를 보여준다.

**답변:**

"1차에서는 `defaultdict` 기반의 효율적 자료구조 설계로 데이터 접근 패턴을 최적화했고, 2차에서는 `SELECT FOR UPDATE`로 동시 접근 패턴을 제어했습니다. 공통점은 **문제의 본질을 먼저 파악하고, 그에 맞는 자료구조/전략을 선택했다**는 것입니다. 1차가 '읽기 최적화'였다면, 2차는 '쓰기 정합성'에 집중한 것입니다."

## 2. AI Native Workflow

### 2.1 "AI를 어떻게 활용했나?"

> 이것이 이 포지션의 핵심 차별점. "AI를 썼다"가 아니라, **"AI에게 엔지니어링 가이드라인을 주입하여 일관된 품질을 확보했다"**는 방법론을 어필한다.

**스토리 흐름:**

**[방법론]** "CLAUDE.md와 7개의 Rules 파일로 AI 에이전트에게 엔지니어링 컨텍스트를 주입했습니다."

| Rules 파일 | 제어 영역 | 효과 |
|-----------|---------|------|
| `coding-style.md` | 불변성 패턴, 타입 힌트, 한국어 에러 메시지 | AI가 생성하는 모든 코드가 동일한 스타일을 따름 |
| `testing.md` | TDD 워크플로우, 80%+ 커버리지 | 테스트 없이 코드를 생성하지 않도록 제약 |
| `agents.md` | 에이전트 오케스트레이션 | planner → tdd-guide → code-reviewer 파이프라인 |
| `security.md` | 시크릿 관리, 입력 검증 | 하드코딩된 키나 검증 누락 방지 |

**[핵심 가치 — 재현 가능성]** "이 구조의 진짜 가치는 **재현 가능성**입니다. 다른 개발자가 동일한 CLAUDE.md를 사용하면, AI가 동일한 코딩 스타일과 아키텍처 판단을 따릅니다. 이것은 팀 단위 AI 활용에서 품질 편차를 줄이는 실질적인 방법론입니다."

**[결과물 검증]** "AI의 출력을 그대로 수용하지 않았습니다."
- `SELECT FOR UPDATE` 도출 과정: AI와 4가지 동시성 전략의 트레이드오프를 논의하고, 수강신청의 높은 충돌률이라는 도메인 특성에 근거하여 최종 결정
- 동시성 테스트: 100개 동시 요청으로 "정확히 1명만 성공"을 DB 레벨까지 검증
- AI가 제안한 코드의 엣지 케이스(시간표 충돌의 경계값, 학점 초과의 동시 요청)를 직접 시나리오화하여 테스트

**[무신사 연결]** "무신사에서 Claude, Cursor 등을 활용한 빠른 개발 문화를 만들고 계신 것으로 알고 있습니다. CLAUDE.md 방식의 '선언적 가이드라인 주입'은 팀 전체의 AI 활용 품질을 표준화하는 데 즉시 기여할 수 있다고 생각합니다."

### 2.2 꼬리 질문 대비: "AI가 잘못된 코드를 생성한 경험은?"

**답변:**

"AI는 낙관적 락을 먼저 제안했습니다. 일반적인 CRUD에서는 충돌 빈도가 낮아 낙관적 락이 합리적이지만, 수강신청은 인기 강좌에 수백 명이 동시에 몰리는 극단적 시나리오입니다. AI에게 '100명이 동시에 1자리 강좌에 신청하는 상황'을 구체적으로 제시하자, 재시도 폭주 문제를 인지하고 비관적 락으로 전환을 제안했습니다."

"이 경험에서 배운 것은, **AI에게 좋은 질문을 하려면 먼저 문제를 정확히 정의해야 한다**는 점입니다. 추상적으로 '동시성 제어 해줘'가 아니라, 구체적인 시나리오와 제약 조건을 제시해야 의미 있는 답을 얻습니다."

## 4. 예상 질문 & 답변 전략

### 4.1 기술 질문

**Q: "커버리지 58%인데, 80% 목표를 달성하지 못한 이유는?"**

"솔직히 시간 제약이 가장 큰 원인입니다. 핵심 경로(인증, 수강신청 CRUD, 100명 동시성 테스트)는 모두 커버했지만, 시드 서비스(0%)와 페이지네이션 경계값 테스트가 미달입니다. 시간이 더 있었다면 Property-based Testing으로 시간표 충돌 로직의 엣지 케이스를 커버했을 것입니다. 중요한 것은, **무엇이 부족한지를 정확히 인지하고 있다**는 점입니다."

**Q: "왜 FastAPI를 선택했나? Spring Boot 경험도 있는데."**

"Python 생태계의 비동기 지원이 결정적이었습니다. `asyncpg` + SQLAlchemy 2.0 async 조합으로 비동기 DB 접근을 네이티브하게 구현할 수 있었습니다. Spring Boot도 충분히 가능했지만, 코딩테스트의 시간 제약 안에서 **빠른 프로토타이핑과 AI 에이전트와의 호환성**을 고려하면 FastAPI가 최적이었습니다. 무신사에서 Spring과 Python을 모두 사용하는 것으로 알고 있는데, 양쪽 모두 실전 경험이 있어 어떤 스택이든 빠르게 적응할 수 있습니다."

**Q: "테스트를 어떻게 설계했나?"**

"두 가지 수준으로 분리했습니다. 단위 테스트(`test_enrollments.py`)에서는 단일 요청 단위로 비즈니스 규칙(정원 초과, 학점 제한, 시간표 충돌, 중복 신청)을 검증하고, 통합 테스트(`test_concurrency.py`)에서는 100개 동시 HTTP 요청을 실제로 보내 응답 코드뿐 아니라 DB의 `enrolled` 값이 정확히 1인지까지 확인했습니다. 이 분리는 Router-Service-Model 계층 아키텍처 덕분에 가능했습니다."

### 4.2 AI 활용 질문

**Q: "AI 없이도 이 시스템을 만들 수 있었나?"**

"네, 만들 수 있습니다. AI의 가치는 '못하는 것을 대신 해주는 것'이 아니라, **'할 수 있는 것을 더 빠르고 일관되게 해주는 것'**입니다. AI가 없었다면 동시성 전략 비교에 더 많은 시간을 썼을 것이고, API 응답 포맷의 일관성을 유지하는 데 더 많은 주의가 필요했을 것입니다. AI 덕분에 절약한 시간을 동시성 테스트 설계와 아키텍처 고도화에 투자할 수 있었습니다."

**Q: "CLAUDE.md 방식의 한계는?"**

"컨텍스트 윈도우의 제약입니다. Rules 파일이 많아지면 AI가 참조할 수 있는 실제 코드의 양이 줄어듭니다. 이 프로젝트에서는 7개 파일로 관리 가능했지만, 대규모 프로젝트에서는 상황에 따라 관련 Rules만 선택적으로 로드하는 전략이 필요합니다. 또한, Rules가 너무 상세하면 AI의 창의적 제안을 제한할 수 있어, **원칙은 명확하되 구현 방식은 열어두는 균형**이 중요합니다."

### 4.3 Culture Fit 질문

**Q: "왜 무신사인가?"**

"세 가지 이유입니다. 첫째, **AI Native 문화입니다.** Claude, Cursor를 일상적으로 활용하는 개발 문화는 제가 CLAUDE.md로 실천해온 방식과 정확히 일치합니다. 둘째, **대규모 트래픽입니다.** 1,700만 사용자, 한정판 발매 트래픽은 제가 코딩테스트에서 다룬 동시성 문제의 실전판입니다. 셋째, **성장 환경입니다.** 시니어 엔지니어와의 페어 프로그래밍과 코드 리뷰를 통해, 지금까지 독학과 프로젝트로 쌓아온 역량의 빈 곳을 채울 수 있다고 기대합니다."

**Q: "팀에서 갈등이 생기면?"**

"Handong Feed를 4명이 개발하면서, 아키텍처 결정에서 의견 충돌이 있었습니다. Spring Boot 모놀리식 vs Polyglot MSA 논쟁이었는데, 각 선택지의 트레이드오프를 표로 정리하고 팀원들과 공유했습니다. 결국 '트래픽 처리는 Spring, AI 연산은 Python'이라는 **기술적 근거에 기반한 합의**를 이끌어냈습니다. 감정이 아니라 데이터와 근거로 설득하는 것이 저의 협업 방식입니다."

**Q: "입사 후 6개월 동안 어떻게 성장할 것인가?"**

"첫 달은 무신사의 코드베이스와 배포 파이프라인을 이해하는 데 집중하겠습니다. 작은 버그 수정이나 기능 개선부터 시작하여 시스템의 맥락을 파악하겠습니다. 이후에는 AI 에이전트 활용 경험을 살려, 팀의 개발 생산성을 높이는 도구나 가이드라인 개선에 기여하고 싶습니다. 6개월 후에는 '이 사람이 없으면 불편하다'는 존재가 되는 것이 목표입니다."

### 4.4 텍사스 교환학생 관련

**Q: "교환학생 경험이 개발에 어떤 영향을 줬나?"**

"기술적으로는 Software Engineering과 Network Security를 원어로 수학하면서, 영어 기술 문서를 읽고 논의하는 역량이 크게 성장했습니다. 프로젝트로는 현지 학생들과 LLM 기반 로컬 뉴스 요약 시스템(LocuSum)을 개발했는데, 문화와 언어가 다른 팀원들과 기술적 합의를 이끌어내는 과정에서 **'명확한 문서화와 구조화된 커뮤니케이션'의 중요성**을 배웠습니다. 이것이 CLAUDE.md 방식의 선언적 가이드라인 접근과도 맞닿아 있다고 생각합니다."

## 5. Reverse Questions (면접관에게 할 질문)

> 면접 마지막에 "궁금한 것 있으신가요?"라고 물을 때. 시니어 엔지니어가 감탄할 만한 질문 3가지.

### Q1: AI 에이전트 가이드라인의 팀 표준화

"저는 이번 코딩테스트에서 CLAUDE.md와 Rules 파일을 통해 AI 에이전트의 코딩 컨벤션과 아키텍처 판단을 제어했습니다. 무신사에서 Claude, Cursor 등을 팀 단위로 활용하실 때, **AI가 생성하는 코드의 품질 편차를 어떻게 관리하고 계신지** 궁금합니다. 팀 공통의 프롬프트 가이드라인이나 에이전트 설정 표준 같은 것이 있으신가요?"

> 의도: AI를 '개인 도구'가 아닌 '팀 인프라'로 보는 시각을 어필. 동시에 무신사의 실제 AI 활용 수준을 파악.

### Q2: 한정판 발매의 동시성 아키텍처

"이번 코딩테스트에서 PostgreSQL `SELECT FOR UPDATE`로 동시성을 제어했는데, 실제 무신사의 한정판 발매처럼 수만 명이 동시에 접근하는 시나리오에서는 **DB 행 잠금만으로는 한계가 있을 것 같습니다.** 혹시 Redis 기반의 사전 필터링이나 대기열 시스템, 또는 다른 아키텍처 패턴을 활용하고 계신지 여쭤봐도 될까요?"

> 의도: 코딩테스트 경험을 실제 비즈니스에 연결하는 사고력 어필. 기술적 호기심과 확장적 사고를 보여줌.

### Q3: 신규 입사자의 온보딩과 첫 기여

"입사 후 첫 달에 가장 효과적으로 기여할 수 있는 방법이 궁금합니다. **코드베이스를 파악하기 위해 권장하시는 첫 번째 작업**(예: 특정 버그 수정, 테스트 추가, 문서화 등)이 있으신가요? 그리고 신규 입사자가 처음 PR을 올릴 때까지 보통 어느 정도의 기간을 기대하시는지 궁금합니다."

> 의도: '빨리 기여하고 싶다'는 적극성. 동시에 겸손하게 팀의 방식을 먼저 배우겠다는 태도.
