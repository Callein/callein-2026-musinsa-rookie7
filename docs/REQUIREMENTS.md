|---|---------|------|------|
| A1 | 인증/인가 | JWT 기반 간단 인증 | 수강신청은 본인 확인 필수. 프론트 연동을 위해 토큰 기반 방식 채택 |

| A4 | 시간표 슬롯 정의 | 월~금, 09:00~18:00, 75분 단위 6개 슬롯 | 실제 대학 시간표 패턴 반영 |

## 2. 데이터 모델 설계

### 2.1 엔티티 관계도

```
Department (학과)
  ├── Professor (교수) : 1:N
  ├── Course (강좌) : 1:N
  └── Student (학생) : 1:N

Course (강좌)
  ├── CourseSchedule (강의 시간표) : 1:N
  ├── Professor : N:1
  └── Enrollment (수강신청) : 1:N

Student (학생)
  └── Enrollment (수강신청) : 1:N

Enrollment (수강신청)
  ├── Student : N:1
  └── Course : N:1
  └── UNIQUE(student_id, course_id)
```

### 2.2 시간표 슬롯

| 교시 | 시간 |
|------|------|
| 1교시 | 09:00 - 10:15 |
| 2교시 | 10:30 - 11:45 |
| 3교시 | 12:00 - 13:15 |
| 4교시 | 13:30 - 14:45 |
| 5교시 | 15:00 - 16:15 |
| 6교시 | 16:30 - 17:45 |

- 요일: 월, 화, 수, 목, 금
- 3학점 강좌: 주 2회 수업
- 2학점 강좌: 주 1회 수업

### 2.3 시간 충돌 판정

두 시간표 슬롯이 충돌하는 조건:
- 같은 요일(day_of_week)이고
- 시간이 겹치는 경우: `start_time_a < end_time_b AND start_time_b < end_time_a`

현재 슬롯 기반 설계에서는 동일 슬롯 비교로 간소화 가능하지만, 향후 유연한 시간 설정을 위해 범위 기반 충돌 판정을 사용합니다.

## 3. 동시성 제어 전략

### 3.1 문제 정의

> "정원이 1명 남은 강좌에 100명이 동시에 신청해도, 정확히 1명만 성공해야 합니다."

이는 전형적인 **재고 차감(inventory decrement)** 문제입니다.

### 3.2 선택한 전략: PostgreSQL `SELECT FOR UPDATE`

**행 레벨 배타적 잠금(Row-Level Exclusive Lock)**을 사용합니다.

```sql
-- 트랜잭션 시작
BEGIN;

-- 해당 강좌 행을 잠금 (다른 트랜잭션은 이 행 수정 시 대기)
SELECT * FROM courses WHERE id = :course_id FOR UPDATE;

-- 비즈니스 규칙 검증
-- 1. enrolled < capacity 확인
-- 2. 학생의 현재 총 학점 + 이 강좌 학점 ≤ 18 확인
-- 3. 시간표 충돌 확인
-- 4. 동일 강좌 중복 신청 확인

-- 통과 시 수강신청 등록
INSERT INTO enrollments (student_id, course_id) VALUES (:sid, :cid);
UPDATE courses SET enrolled = enrolled + 1 WHERE id = :course_id;

COMMIT;
```

### 3.3 트랜잭션 격리 수준

PostgreSQL의 기본 격리 수준인 **Read Committed**를 사용합니다.

Read Committed 환경에서도 `SELECT FOR UPDATE`는 다음을 보장합니다:
- 잠금 획득 시점에 **최신 커밋된 데이터**를 읽음
- 다른 트랜잭션이 같은 행을 수정하려 하면 **대기(wait)**
- 잠금을 보유한 트랜잭션이 커밋/롤백할 때까지 **직렬화**

따라서 성능 비용이 높은 Serializable 격리 수준까지는 필요하지 않습니다. Read Committed + 행 잠금 조합이 성능과 정합성의 최적 균형점입니다.

### 3.4 데드락 방지

**API 설계 수준에서 방지합니다:**

- 수강신청 API는 **한 번에 한 과목만** 신청 가능
- 따라서 단일 트랜잭션에서 하나의 Course 행만 잠금
- 여러 행을 잠그지 않으므로 순환 대기(circular wait) 불가 → **데드락 발생 불가**

### 3.5 동일 학생의 동시 요청 처리

같은 학생이 동시에 여러 요청을 보내는 경우:
- `UNIQUE(student_id, course_id)` 제약으로 동일 강좌 중복 방지
- 학점/시간 충돌 검증은 Course 행 잠금 이후 트랜잭션 내에서 수행
- 서로 다른 강좌를 동시에 신청하는 경우, 각각 독립적인 행 잠금이므로 정합성 유지

### 3.6 에러 응답 정의

수강신청 실패 시 HTTP 상태 코드:

| 실패 사유 | HTTP 상태 코드 | 에러 메시지 |
|----------|--------------|------------|
| 정원 초과 | 409 Conflict | "해당 강좌의 정원이 초과되었습니다." |
| 학점 초과 (18학점) | 409 Conflict | "최대 수강 학점(18학점)을 초과합니다." |
| 시간표 충돌 | 409 Conflict | "기존 수강 강좌와 시간이 충돌합니다." |
| 동일 강좌 중복 | 409 Conflict | "이미 수강 신청한 강좌입니다." |
| 강좌 미존재 | 404 Not Found | "해당 강좌를 찾을 수 없습니다." |
| 인증 실패 | 401 Unauthorized | "유효하지 않은 인증 토큰입니다." |

> 정원 초과, 학점 초과, 시간 충돌 등은 **리소스 상태 경합(resource contention)**으로 인한 실패이므로 `409 Conflict`가 적합합니다. `400 Bad Request`는 요청 형식 오류에 사용합니다.

### 3.7 대안 검토

| 전략 | 장점 | 단점 | 채택 여부 |
|------|------|------|----------|
| SELECT FOR UPDATE | DB 레벨 보장, 검증됨 | PostgreSQL 의존 | ✅ 채택 |
| 낙관적 잠금 (version) | 충돌 적을 때 성능 좋음 | 수강신청 특성상 충돌 빈번 → 재시도 폭주 | ❌ |
| Application Lock (asyncio.Lock) | 단일 프로세스에서 간단 | 다중 프로세스/서버 환경 불가 | ❌ |
| Redis 분산 락 | 다중 서버 지원 | 추가 인프라 필요, 복잡도 증가 | ❌ |

## 4. 인증/인가 설계

### 4.1 결정

- **방식**: JWT (JSON Web Token) 기반 Bearer 인증
- **근거**:
  - 프론트엔드 연동 시 가장 일반적인 방식
  - Stateless → 서버 확장 용이
  - 수강신청 시 본인 확인이 반드시 필요

### 4.2 인증 흐름

```
1. POST /api/auth/login (학번 + 비밀번호)
   → JWT 토큰 반환

2. 보호된 API 요청 시 Header에 토큰 포함
   Authorization: Bearer <token>

3. 서버에서 토큰 검증 → 학생 식별
```

### 4.3 인증 범위

| API | 인증 필요 | 이유 |
|-----|----------|------|
| 조회 API (학생/강좌/교수) | X | 프론트팀 연동 편의 |
| POST /api/enrollments | O | 수강신청은 본인만 가능 |
| DELETE /api/enrollments/{id} | O | 수강취소는 본인만 가능 |
| GET /api/enrollments/me/schedule | O | 내 시간표는 본인만 조회 |

## 5. API 설계 원칙

### 5.1 URL 구조

- 모든 API는 `/api` 프리픽스 사용
- 복수형 명사로 리소스 표현: `/students`, `/courses`, `/professors`, `/enrollments`
- 헬스체크는 프리픽스 없이 `/health`

### 5.2 응답 형식

**목록 조회 (페이지네이션 포함):**
```json
{
  "success": true,
  "data": [...],
  "meta": {
    "total": 10000,
    "page": 1,
    "limit": 20,
    "total_pages": 500
  }
}
```

**단일 리소스 조회:**
```json
{
  "id": 1,
  "name": "자료구조",
  "credits": 3,
  ...
}
```

**에러 응답:**
```json
{
  "success": false,
  "error": "사용자 친화적 에러 메시지"
}
```

### 5.3 페이지네이션

- 기본 페이지 크기: 20
- 최대 페이지 크기: 100
- offset 기반 (page, limit 파라미터)

## 6. 데이터 생성 전략

### 6.1 규모

| 항목 | 수량 | 생성 방식 |
|------|------|----------|
| 학과 | 12개 | 실제 대학 학과명 토큰 목록 |
| 교수 | ~120명 | 학과당 8~12명, 한국 성+이름 조합 |
| 강좌 | 500개+ | 학과별 과목명 토큰 + 분반, 시간표 자동 배정 |
| 학생 | 10,000명 | 한국 성+이름 조합, 학번 자동 생성 |

### 6.2 멱등성

- 서버 시작 시 `departments` 테이블의 레코드 수를 확인
- 이미 데이터가 존재하면 시드 생성을 스킵
- 서버를 재시작해도 데이터가 중복 생성되지 않음

### 6.3 현실성

- 학과명: 컴퓨터공학과, 전자공학과, 경영학과 등 실제 대학 학과
- 과목명: 자료구조, 알고리즘, 미시경제학 등 실제 대학 과목
- 학생 이름: 한국 성(30개) + 이름(50개) 조합
- 교수 이름: 동일 방식, 중복 방지
- 비밀번호: 모든 학생 "password" (테스트 편의)

### 6.4 데이터 정합성

- 교수의 시간표 충돌 방지: 동일 교수에게 같은 시간대 강좌 배정 불가
- 고정 시드(seed=42) 사용으로 재현 가능한 데이터 생성
- batch insert(1000건 단위)로 1분 이내 완료 보장
